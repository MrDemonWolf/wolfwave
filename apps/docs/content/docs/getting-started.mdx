<span className="docs-eyebrow">Build from source</span>

This is for developers building WolfWave from source. Just want to use the app? Grab the [Installation](./installation) guide instead.

## Before you start

You'll need:

- **macOS 26.0+** (Tahoe)
- **Apple Silicon** (M1 or later)
- **Xcode 16.0+**
- **Swift 5.9+**
- **Command Line Tools.** Run `xcode-select --install` if you haven't.

## Clone the repo

```bash
git clone https://github.com/MrDemonWolf/WolfWave.git
cd WolfWave
```

## Configure API keys

**Copy the example config:**

```bash
cp apps/native/WolfWave/Config.xcconfig.example apps/native/WolfWave/Config.xcconfig
```

**Open `apps/native/WolfWave/Config.xcconfig`** and set your keys:

```
TWITCH_CLIENT_ID = your_twitch_client_id_here
DISCORD_CLIENT_ID = your_discord_client_id_here

# Optional. Fork/whitelabel branding overrides. Leave as-is for the upstream defaults.
DOCS_URL = https://mrdemonwolf.github.io/wolfwave
COMMUNITY_DISCORD_URL = https://mrdwolf.net/discord
COPYRIGHT_HOLDER = MrDemonWolf, Inc.
```

<Callout type="info">
  Get a Twitch Client ID at
  [dev.twitch.tv/console/apps](https://dev.twitch.tv/console/apps). Get a Discord
  Application ID at
  [discord.com/developers/applications](https://discord.com/developers/applications).
</Callout>

<Callout type="warn">
  `Config.xcconfig` is gitignored. Each contributor uses their own.
</Callout>

<Callout type="info">
  **Forking WolfWave?** Override `DOCS_URL`, `COMMUNITY_DISCORD_URL`, and
  `COPYRIGHT_HOLDER` to point at your own site, community, and legal entity.
  Each falls back to the upstream MrDemonWolf default if left blank. To rename
  the app itself, edit `CFBundleDisplayName` in `Info.plist` (or set
  `PRODUCT_NAME` in an xcconfig overlay).
</Callout>

## Open the project

```bash
make open-xcode
# or
open apps/native/WolfWave.xcodeproj
```

## Build and run

**Hit ⌘R in Xcode.** That's it.

## Production build

**For a release build with a DMG:**

```bash
make prod-build
```

The DMG lands at `builds/WolfWave-{version}-arm64.dmg`. Have a Developer ID Application certificate in your Keychain? The app gets re-signed for distribution as part of the build.

**To build and drop straight into `/Applications`:**

```bash
make prod-install
```

## Notarization

**Notarize the DMG** so Gatekeeper trusts it:

```bash
APPLE_ID=you@example.com \
APPLE_TEAM_ID=YOUR_TEAM_ID \
APPLE_APP_PASSWORD=xxxx-xxxx-xxxx-xxxx \
make notarize
```

<Callout type="info">
  Generate an app-specific password at
  [appleid.apple.com](https://appleid.apple.com/account/manage) under **Sign-In and
  Security → App-Specific Passwords**.
</Callout>

## CI/CD

**Push a version tag** to kick off the GitHub Actions release workflow. It builds, signs, notarizes, and staples the app, generates the Sparkle appcast, and uploads the DMG (plus `appcast.xml` and the release-notes HTML) to a draft GitHub Release:

```bash
git tag v2.0.0
git push origin v2.0.0
```

**Then review and publish.** Open the draft release, confirm the notarized DMG and notes look right, and click Publish. The Homebrew cask updates on its own once the release is live.

Heads up: the `TWITCH_CLIENT_ID` and `DISCORD_CLIENT_ID` secrets must be set in your repo's Actions secrets. The full release checklist lives in the [Development Guide](./development) and [PUBLISH.md](https://github.com/MrDemonWolf/WolfWave/blob/main/PUBLISH.md).
